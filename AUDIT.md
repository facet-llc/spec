# Audit Records (Ed25519)

Every settled Facet transaction returns a signed audit record. The signature is the proof an agent or auditor uses to verify, after the fact, that a specific transaction settled at a specific merchant, for a specific amount, between a specific agent and merchant identity.

This document specifies how the audit record is canonicalized, signed, and verified. It is normative for any Facet-compliant merchant.

## Record shape

Per [`schemas/v1.audit.json`](./schemas/v1.audit.json):

```json
{
  "txn_id": "txn_abc123",
  "iat": 1730000000,
  "merchant": "did:facet:merchant-1",
  "agent": "did:facet:agent-1",
  "amount": "250.00",
  "currency": "USDC",
  "tx_hash": "0xabc...",
  "sig": "<base64url(Ed25519(canonical(record)))>",
  "kid": "merchant-2026-05"
}
```

Required fields: `txn_id`, `iat`, `merchant`, `agent`, `sig`, `kid`. Optional fields: `amount`, `currency`, `tx_hash`.

## Canonicalization

The bytes that get signed are produced this way:

1. Take the record object.
2. Remove the `sig` and `kid` fields.
3. Serialize the remaining object as JSON with these rules:
   - UTF-8 encoding
   - Sort top-level keys lexicographically
   - No whitespace between tokens (no indentation, no separator spaces)
   - Strings escaped per RFC 8259 (no extra escapes)
4. The resulting byte sequence is the message that goes through Ed25519.

This is a subset of [RFC 8785](https://www.rfc-editor.org/rfc/rfc8785) (JSON Canonicalization Scheme). v0.1 only supports flat top-level objects so the simpler subset suffices. v0.2 will widen to full RFC 8785 when nested objects appear in audit records.

## Signing

The merchant's signing key is an Ed25519 keypair. The public key is published as a JWK in the merchant's JWKS (resolvable from the merchant DID document or `<merchant-iss>/.well-known/jwks.json`).

JWK shape:

```json
{
  "kty": "OKP",
  "crv": "Ed25519",
  "x": "<base64url(public_key_bytes)>",
  "kid": "merchant-2026-05",
  "alg": "EdDSA",
  "use": "sig"
}
```

The merchant signs the canonicalized bytes with their Ed25519 private key, base64url-encodes the resulting 64-byte signature, and sets `sig`. The `kid` field MUST identify the JWK used.

## Verification

Verifiers MUST:

1. Extract `sig` and `kid` from the record.
2. Re-canonicalize the remaining record per the rules above.
3. Resolve the merchant's JWKS and find the JWK with the matching `kid`.
4. Validate that `kty=OKP`, `crv=Ed25519`, `alg=EdDSA`.
5. Verify the Ed25519 signature over the canonical bytes.
6. Reject if any check fails. No silent fallbacks, no key fuzzy-matching.

The reference TypeScript implementation lives in [`sdks/typescript/src/audit.ts`](./sdks/typescript/src/audit.ts) and exposes `verifyAuditRecord(record, options)`.

## Conformance vectors

Test vectors for audit-record verifiers live in [`test-vectors/audit/`](./test-vectors/audit/). v0.1 ships:

| Vector | Scenario |
|---|---|
| `01-valid-audit.json` | Happy path. Verifier returns `verified: true`. |
| `02-tampered-amount.json` | `amount` field changed after signing. Reject. |
| `03-wrong-key.json` | Signed with key A, JWKS contains only key B. Reject. |
| `04-missing-required-field.json` | `agent` field stripped. Verifier MUST reject before checking signature. |

Generated by `test-vectors/generate_audit.py`. Same fresh-keypair pattern as the KYAPay vectors: private key is in-memory only, never persists.

## Why Ed25519 and not ES256

KYAPay JWTs use ES256 (ECDSA P-256). Audit records use Ed25519. Different primitives because:

1. Ed25519 produces deterministic signatures (same message + same key = same signature). ES256 doesn't. For audit records, deterministic signatures simplify replay verification and cross-implementation interop testing.
2. Ed25519 is faster to verify (~5× than ES256 at typical key sizes), which matters when an agent fans out to verify hundreds of audit records as proof of a session's transaction history.
3. Ed25519 has smaller signatures (64 bytes vs ~70 for ES256), which adds up across audit logs.

The protocol requires merchants support both: ES256 inbound (verifying agent KYAPay JWTs) and Ed25519 outbound (signing audit records).

## What this doc does NOT cover (yet)

- Multi-signature audit records (v0.2)
- Audit-record chaining (Merkle proofs across a session) (v0.2)
- Cross-issuer audit federation (v0.3)
- Hardware-backed signing keys (HSM, TEE) (v1.0)
